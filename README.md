# sendgrid — an agent-facing CLI & MCP server for the SendGrid v3 API

A single Rust binary, `sendgrid`, that gives AI agents (and humans) validated,
safety-gated access to **all 391 SendGrid v3 API operations** — every endpoint
across 46 API groups, not a hand-picked subset.

The whole surface is **autogenerated** from the official
[`twilio/sendgrid-oai`](https://github.com/twilio/sendgrid-oai) OpenAPI specs, so
it tracks the real API rather than a curated wrapper. Two surfaces sit over one
dispatch chokepoint:

- **CLI** — `sendgrid <domain> <subgroup> <verb>-<noun>`
  (e.g. `sendgrid mail send send-mail --body @msg.json`).
- **MCP server** — `sendgrid mcp` exposes the entire API through **3 meta-tools**
  (`search_operations` → `describe_operation` → `invoke_operation`) so an agent
  discovers and calls any operation without 391 tool definitions bloating its
  context.

Both paths funnel through the same `execute()` function in `sendgrid-core`, so
validation, the side-effect policy, secret redaction, region routing, pagination,
and retries behave identically whether a human or an agent makes the call.

## Why this design

- **All APIs, not a subset.** Agents need the long tail (suppressions, IP pools,
  subusers, data erasure), not just "send an email". All 391 ops are reachable.
- **Autogenerated, drift-aware.** The command tree, MCP tools, JSON-Schema
  validation, and safety metadata are all derived from the vendored specs via
  `cargo xtask codegen`. Spec drift is a re-generation, not a rewrite.
- **One safety chokepoint.** Side-effect classification, secret redaction, and
  impersonation governance are enforced once, in `execute()` — there is no second
  code path to forget to harden.
- **Self-contained binary.** Pure-Rust [ring](https://github.com/rustls/rustls)
  TLS with bundled webpki roots and the API contract embedded at compile time —
  no system OpenSSL, no cert store, no sidecar files.

## Install

Requires **Rust ≥ 1.96** (the workspace pins `rust-version = "1.96"`) and
**edition 2024**.

From the workspace root:

```sh
# Install the `sendgrid` binary onto your PATH:
cargo install --path crates/cli

# …or just build a release binary in-tree:
cargo build --release -p sendgrid-cli
# → ./target/release/sendgrid   (~6 MiB, self-contained)
```

The produced binary is named **`sendgrid`**. It statically embeds the operation
contract (the generated IR + per-op JSON Schemas) and links ring TLS with bundled
Mozilla roots, so it runs in a bare container with no extra files or system
certificate store.

## 60-second quickstart

```sh
# 1. Set your API key (preferred over --api-key, which can leak via shell history).
export SENDGRID_API_KEY="SG.xxxxxxxx.yyyyyyyy"

# 2. Verify setup: key present + well-formed, region reachable, live scopes call.
sendgrid auth doctor

# 3. Preview a send WITHOUT sending it (works on any operation).
#    Note: global flags like --dry-run go BEFORE the domain/subcommand.
sendgrid --dry-run mail send send-mail --body '{
  "personalizations": [{ "to": [{ "email": "to@example.com" }] }],
  "from": { "email": "from@example.com" },
  "subject": "hello",
  "content": [{ "type": "text/plain", "value": "hi from sendgrid-cli" }]
}'

# 4. A real read call (Get a user's account information).
sendgrid account user list-account
```

`auth doctor` prints a JSON report and exits non-zero if anything is wrong, so it
doubles as a CI / agent pre-flight check:

```jsonc
{
  "api_key": { "present": true, "well_formed": true, ... },
  "checks": [
    { "name": "api_key_present",  "status": "pass", ... },
    { "name": "region_available", "status": "pass", ... },
    { "name": "scopes_call",      "status": "pass", ... }
  ],
  "ok": true,
  "region": "global"
}
```

## The command model

The CLI tree is built at runtime from the operation registry. The everyday shape
is:

```
sendgrid <domain> <subgroup> <verb>-<noun> [--flags]
```

```sh
sendgrid mail send send-mail            --body @msg.json
sendgrid marketing contacts get-contact --id 123
sendgrid account user list-account
```

> **Flag placement matters.** Global flags (`--dry-run`, `--output`, `--query`,
> `--all`, `--limit`, `--region`, `--allow`, `--on-behalf-of`, `--api-key`,
> `--include-legacy`) are **root-level** and must appear **before** the domain.
> Operation parameters (`--id`, `--body`, …) are leaf flags and go **after** the
> leaf command. So: `sendgrid --dry-run mail send send-mail --body @msg.json`.

There are 14 visible top-level **domains** (`account`, `branding`, `compliance`,
`ips`, `mail`, `marketing`, `security`, `senders`, `settings`, `stats`,
`suppressions`, `templates`, `validation`, `webhooks`), plus the built-ins
`search`, `mcp`, and `auth`.

**One nuance:** operations with longer names nest one level deeper instead of
flattening. For example `DeleteContact` is the flat leaf
`marketing contacts delete-contact`, while `DeleteContactIdentifier` nests as
`marketing contacts delete contact-identifier`. You never have to memorize this —
use discovery instead:

```sh
# Full-text search across id, summary, tags, domain, and HTTP path:
sendgrid search contact list
sendgrid search "send email"

# Drill down with --help at any level; it lists subcommands and flags:
sendgrid marketing --help
sendgrid marketing contacts --help
sendgrid mail send send-mail --help
```

Each `search` hit shows the operation id, HTTP method/path, summary, and where it
lives in the command tree. To run a leaf, address it as `<groups…> <verb>-<noun>`
— the last two path words joined by a hyphen (so the op shown at
`marketing contacts delete contact` runs as
`sendgrid marketing contacts delete-contact`). When in doubt, confirm the exact
spelling with `--help` on the parent group.

Legacy/superseded operations (e.g. the v1 marketing campaign API) are hidden by
default. Add `--include-legacy` to expose them in both the tree and `search`.

### Request bodies (`--body`)

Operations with a request body take a single `--body` flag that accepts three
forms:

```sh
sendgrid mail send send-mail --body '{"personalizations":[...]}'   # inline JSON
sendgrid mail send send-mail --body @message.json                  # @path to a file
cat message.json | sendgrid mail send send-mail --body -           # - reads stdin
```

Bodies are validated against the operation's JSON Schema **and** a set of
cross-field rules the schema can't express (e.g. SendMail requires `content`
*or* `template_id`; `reply_to` and `reply_to_list` are mutually exclusive). You
get an actionable, JSON-pointer-style error *before* anything is sent.

Path / query / header parameters are ordinary `--flags` named exactly as the spec
names them (see each leaf's `--help`).

### Output & `--query`

```
--output json|table|csv|ndjson   # default: json
--query <PATH>                   # jq-lite selector over the response `data`
```

```sh
sendgrid --output table marketing lists list marketing-list
sendgrid --output ndjson --query 'result[].id' marketing lists list marketing-list
```

`json` returns the full result envelope (pretty on a TTY, compact otherwise);
`table`/`csv`/`ndjson` render the `data` payload. `--query` is a small jq-style
field selector (e.g. `result[].id`), not a full jq.

### Pagination

```
--all                # auto-follow pages up to the caps below
--limit <N>          # cap total accumulated items under --all
--offset <N>         # starting offset (offset-paginated ops)
--page-token <TOKEN> # explicit page token (page-token ops)
```

```sh
sendgrid --all --limit 500 marketing lists list marketing-list
```

The runtime knows each operation's pagination style (cursor key, page number,
offset, page token, …) from the IR, so `--all` follows the right field
automatically. When it stops at a cap, the result envelope's `next` field carries
the continuation cursor.

### Regions

```
--region global|eu   # default: global   (`us` is accepted as an alias of global)
```

SendGrid has exactly two data regions. **14 of the 46 API groups are
global-only** (they declare no EU host). Requesting `--region eu` for one of those
**fails closed** rather than silently routing your data to the global region —
re-run with `--region global` if that is acceptable. See
[Known limitations](docs/limitations.md#eu-region-fails-closed-for-global-only-groups).

### Dry-run everything

`--dry-run` builds, validates, and returns the exact request that *would* be sent
(URL, method, redacted headers, body) without sending it. It works on every
operation and bypasses the side-effect policy, so it is the safe way to inspect a
destructive or send-class call before committing.

## MCP setup

Run the server over stdio:

```sh
sendgrid mcp
```

Point your MCP client at the binary. The key is supplied via the environment, so
it is **never** an MCP tool parameter:

```jsonc
{
  "mcpServers": {
    "sendgrid": {
      "command": "sendgrid",
      "args": ["mcp"],
      "env": { "SENDGRID_API_KEY": "SG.xxxxxxxx.yyyyyyyy" }
    }
  }
}
```

> Use an absolute path for `command` (e.g. `/usr/local/bin/sendgrid` or the
> `cargo install` location) if `sendgrid` is not on your client's PATH.

### The 3-tool workflow

The agent sees only three tools and an instructions string that teaches the loop:

1. **`search_operations`** `{ query, [domain], [tags], [side_effect], [method], [limit] }`
   → ranked, metadata-only hits (`id`, `summary`, `method`, `path`, `side_effect`,
   `tags`). Free-text; start here.
2. **`describe_operation`** `{ id, [expand: minimal|full] }`
   → params, required fields, a compact body example, and cross-field constraints
   (`minimal`, default) or the full request-body JSON Schema (`full`).
3. **`invoke_operation`** `{ id, [path_params], [query], [headers], [body], [dry_run], [confirm] }`
   → executes through the same `execute()` chokepoint and returns the uniform
   result envelope (`status`, `side_effect`, `data | error`, `request_preview`,
   `next`, `warnings`).

Operation ids are `sg_<domain>_<subgroup>_<operationId>` (the subgroup is dropped
when it equals the domain), e.g. `sg_mail_send_SendMail`.

### Hardening the MCP server

The default policy **allows every side-effect class**. For an autonomous agent,
launch the server restricted and let it escalate only when needed:

```jsonc
{
  "mcpServers": {
    "sendgrid": {
      "command": "sendgrid",
      "args": ["--allow", "read", "mcp"],
      "env": { "SENDGRID_API_KEY": "SG.xxxxxxxx.yyyyyyyy" }
    }
  }
}
```

With `--allow read`, write/destructive/send calls return `E_POLICY_DENIED` and the
agent is steered toward `dry_run: true` previews. See the full
[Safety model](docs/safety.md) for the recommended presets, the always-on
invariants, and why `confirm` is **not** a security control.

`--expose-tag <TAG>` / `--expose-op <OP_ID>` (repeatable) optionally promote
specific operations to first-class MCP tools that take their parameters directly
— still routed through the same safety pipeline.

## Authentication

| Source | Notes |
|---|---|
| `SENDGRID_API_KEY` env var | **Preferred.** How the MCP config and quickstart set the key. |
| `--api-key <KEY>` flag | Discouraged (leaks via shell history / process listings); the CLI prints a warning when used. |

The key is resolved once, redacted everywhere, and **never** logged, echoed, or
returned — including the live key in a `CreateApiKey` response. `auth whoami`
shows a non-reversible fingerprint, never the key itself.

Config-file profiles (multiple accounts, keychain/`key_command` key sources) are
**designed but not yet implemented** — use the env var / flags today. See
[Known limitations](docs/limitations.md#config-file-profiles-are-not-implemented).
